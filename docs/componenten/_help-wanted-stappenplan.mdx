# Stappenplan: Help wanted

Volg dit stappenplan om een component van alle checks te voorzien die nodig zijn voor de ['Help wanted' status](https://github.com/orgs/nl-design-system/projects/27/views/1) van het [Estafettemodel](https://www.nldesignsystem.nl/componenten/definition-of-done).

## Naam bepalen

Begin met het bepalen van een naam.

Baseer, indien mogelijk, de naam van het component op herkenbare termen uit webstandaarden. Gebruik liever termen uit HTML, CSS, SVG en WAI-ARIA dan termen uit 'UI frameworks' die momenteel populair zijn. We schrijven componenten over het algemeen in het enkelvoud.

Kijk voor inspiratie bij:

- [Component gallery](https://component.gallery)
- [UI guideline](https://www.uiguideline.com/components)
- [NL Design System componenten](https://www.nldesignsystem.nl/componenten/)

Zijn er meerdere namen voor het component? Kies dan de meest gangbare. De andere namen kun je later toevoegen aan de documentatie.

## Discussion starten

Doel: Een plek hebben om input te verzamelen.

We gebruiken [Github 'Discussions' van NL Design System](https://github.com/orgs/nl-design-system/discussions) om input te verzamelen.

1. Start een [Discussion](https://github.com/orgs/nl-design-system/discussions/new?category=component-suggestions) voor het component.
2. Vul als titel in: `Component: {naam-component}`.
3. Vul als beschrijving onderstaande template in en start de Discussion.

```
## Naam

## Doel

## Ook bekend als

## Gerelateerde componenten

## Onderzoek
```

## Rationale opstellen

Doel: Zorgen dat we als community op 1 lijn zitten wat betreft wat het component wel en niet is. Daarnaast willen we inzichtelijk maken of er gerelateerde componenten zijn.

Een rationale bestaat in de basis uit een 'naam' en een 'doel'.

### Naam

De naam heb je al. Noteer deze als volgt:

```
## Naam
{naam-component}
```

### Doel

Het doel (purpose) van het component beschijft kort waar het component voor dient. Houd deze kort en bondig.

Kijk ter inspiratie bij:

- https://design-system.service.gov.uk
- https://designsystem.digital.gov
- https://spectrum.adobe.com
- https://carbondesignsystem.com
- https://cedar.rei.com

We proberen de schrijfwijze van het doel zoveel mogelijk te beginnen met een werkwoord en de term 'gebruiker' te vermijden. Bekijk bestaande [Discussions](https://github.com/orgs/nl-design-system/discussions/categories/component-suggestions) als voorbeeld.

Noteer het doel als volgt:

```
## Doel
{doel-component}
```

### Ook bekend als

Zijn er andere namen voor dit component? En is de kans groot dat men op die naam gaat zoeken? Noteer deze dan als volgt:

```
## Ook bekend als
X, Y, of Z.
```

Zijn er geen andere namen? Laat dan dit onderdeel weg.

### Gerelateerde componenten

Het kan voorkomen dat andere componenten sterk gerelateerd zijn aan dit component, zoals bijvoorbeeld Button en Button Group. In dat geval is het handig om deze componenten naar elkaar te laten verwijzen, indien mogelijk via een link. Noteer gerelateerde componenten als volgt:

```
## Gerelateerde componenten
X, Y, of Z.
```

Zijn er geen gerelateerde componenten? Laat dan dit onderdeel weg.

### Community om hulp vragen

Om meer te leren over het component vragen we de community om hulp. Deze oproep doen we via de Discussion en Slack.

#### Afbeelding maken

Om de community letterlijk een beeld te geven van het component voegen we een afbeelding toe aan de oproep. Deze kun je maken in [Figma 'Schetsboek'](https://www.figma.com/file/fy08SZpZmqx6ljLwvA3Woe/NLDS---Schetsboek?type=design&node-id=439-969&mode=design).

#### Oproep via Discussion

Plaats de oproep bovenaan in de beschrijving van de Discussion.

1. Voeg de afbeelding toe.
2. Wijzig de 'alt text' die automatisch door Github wordt toegevoegd. Schrijf bijvoorbeeld:

- `verschillende vormen van het {naam-component} component`
- `visuele weergave van het {naam-component} component`

3. Voeg onderstaande tekst toe, vul aan waar nodig en 'Update' de Discussion.

```
We willen graag meer leren over het {naam-component} component. In welke vorm, of vormen, wordt deze momenteel ingezet en toegepast binnen jouw organisatie?

Deel hieronder een link naar Figma, Storybook of screenshots. Met deze informatie kunnen we bewijzen dat het component onderdeel moet worden van de NL Design System component library.

Daarnaast wordt het delen van onderzoek gewaardeerd. Dus heb je zelf onderzoek gedaan en zijn er inzichten voor dit component? Dan horen we dit ook graag!

[Link naar bericht in Slack]({link-naar-Slack-bericht})

---
```

De link van het Slack bericht genereer je hieronder.

#### Oproep via Slack

Plaats de oproep in het Slack channel [#nl-design-system](https://codefornl.slack.com/archives/C7KDQRXMZ).

1. Voeg de afbeelding toe aan je bericht.
2. Voeg ook hier de 'alt text' toe. [Alt text toevoegen aan een Slack bericht](https://slack.com/help/articles/4403914924435-Add-descriptions-to-images)
3. Voeg onderstaande tekst toe, vul aan waar nodig en verstuur je bericht.

```
Hey community ðŸ‘‹

We willen graag meer leren over het {naam-component} component. In welke vorm, of vormen, wordt deze momenteel ingezet en toegepast binnen jouw organisatie?

Deel een link naar Figma, Storybook of screenshots in de [Github Discussion voor het {naam-component} component]({link-naar-discussion}), of in een thread onder dit bericht. Met deze informatie kunnen we bewijzen dat het component onderdeel moet worden van de NL Design System component library.

Daarnaast wordt het delen van onderzoek gewaardeerd. Dus heb je zelf onderzoek gedaan en zijn er inzichten voor dit component? Dan horen we dit ook graag!
```

Vergeet niet de link naar het Slack bericht toe te voegen aan de oproep in de Discussion.

### Varianten vastgeleggen

Doel: Alleen varianten die geen snowflakes zijn, worden vastgelegd. Bovendien is duidelijk waarvoor elke variant bedoeld is.

Deze stap heb je waarschijnlijk al genomen wanneer je alle checks onder 'Screenshots beschikbaar voor diverse huisstijlen' hebt afgevinkt. Maar als je twijfelt over de rationale van een variant, kun je eventueel de betreffende organisatie om aanvullende toelichting vragen.

**ðŸš© Checkpoint**

- [ ] Varianten die regelmatig voorkomen zijn vastgelegd

## Onderzoek toevoegen

Doel: We kunnen aantonen dat dit component goed werkt voor eindgebruikers.

Zijn er inzichten over het component beschikbaar omdat dit een rol heeft gespeeld bij gebruikersonderzoek? Super! Voeg die inzichten toe. Daarnaast kun je 'desk-research' doen om inzichten en 'best-practices' te verzamelen.

Kijk bijvoorbeeld bij:
