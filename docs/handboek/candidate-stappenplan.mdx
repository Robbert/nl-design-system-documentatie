# Stappenplan: Candidate

Dit component is opgenomen in de `@nl-design-system-unstable` organisatie. Het kernteam verwacht dat het component uiteindelijk een Hall of Fame status krijgt, maar kan nog niet garanderen dat API's niet wijzigen of er onverhoopt een ander vergelijkbaar component voor in de plaats komt. Het component moet mogelijk nog extra goed getest en stabiel gemaakt worden. Omdat het component door het kernteam onderhouden wordt voldoet het, naast de meeste Community DoD eisen, al wel aan veel meer criteria voor stabiel hergebruik.

## Ingezet door twee of meer organisaties met verschillende huisstijlen.

### **Doel**

Het component toont aan dat het met meerdere huisstijlen overweg kan. 

Controleer of er voldoende properties middels tokens aan te passen zijn om aan de wensen te voldoen van de huisstijlen van de betrokken organisaties.

### Voorbeeld

Via de tokens ```{button.primary.default}``` en ```{button.radius}``` kunnen twee organisaties hun specifieke huisstijl doorvoeren op hetzelfde component.

- Utrecht heeft een blauwe actie kleur en een border-radius van 2px
- Amsterdam heeft een rode actie kleur en een border-radius van 8px

**🚩 Checkpoint**: > 2 Thema’s

## Gespiegelde werkelijkheid tussen Figma en code

### **Doel**
De NL Design System Figma bibliotheek komt overeen met de implementatie van het component in code wat betreft varianten, states, design tokens en naamgeving.

### Voorbeeld:
Een component kan gebruik maken van enums (selectie lijst in Figma), booleans (toggle in Figma), en strings (inputfield in Figma). Hoe je de eigenschappen (properties) noemt en hoe je de eigenschap aanstuurt, kun je gelijk trekken tussen Figma en de Code implementatie.
